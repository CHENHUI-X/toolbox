# Hermes Vault — 用户保险柜系统

用户不需要自己记笔记，全靠我帮他存。文件在 `~/.hermes/vault/` 目录下。

## 文件结构

```
~/.hermes/vault/
├── 🔐 密码和密钥.md       ← 密码、API Key、Token 等敏感信息
├── 🖥️ 服务器配置.md        ← IP、端口、服务路径、节点信息
├── 📋 已完成的任务.md       ← 用户完成过的事情记录
├── ⚙️ 偏好设置.md           ← 用户喜好、习惯、沟通风格
└── 💡 小技巧.md             ← 实用命令、小发现
```

## 用法

- **存**：用户说"帮我记住 xxx" → 用 patch 或 write_file 写入对应文件
- **查**：用户问"xxx 是多少" → 用 read_file 读取对应文件，提取信息
- **这些都是纯文本 .md 文件，永久存储，不占 memory 的 token 配额**
- **用户也可以直接用电脑打开看/编辑**
- **参考模板**：`skill_view(name='hermes-vault', file_path='references/vault-template.md')` 可获取文件模板（含各分类的示范结构）

## 注意

- 不要往 memory 里塞太多具体数据（密码、配置细节），放 vault 文件
- memory 只存用户的偏好、习惯这些轻量信息

## 会话习惯（本用户的偏好）

此用户在 vault 使用过程中暴露了以下偏好，每次会话都应遵循：

### 通知风格
- **没变化就别说话**：检查脚本（cron 更新检测、健康检查）如果一切正常，应当静默退出（exit 0, stdout 为空）
- **只有真有事才通知**：更新可用、服务挂了、重启成功 — 这些才发消息
- 不要发"一切正常"的日常报告，用户觉得"有点乱"

### API Key 处理原则
- 如果 config 文件中的 key 被 mask 显示（如 `sk-nxA...85UM`），如实记录被 mask 后的值 + 注明在哪个配置文件
- 如果用户要求存完整 key，尝试从 auth.json、环境变量、进程 environ 中查找
- 如果找不到完整 key，保存配置文件中显示的值并告知用户情况
- 存 key 时同时记下：提供商名称、API 地址、模型名、上下文长度、配置文件名和行号

### Hermes 环境红线
- **绝对不要碰** Hermes 的 venv、pip、Python 解释器
- 更新只做 `git pull` 拉代码，不执行 `pip install` 或 venv 操作
- 如果 venv 坏了，告知用户并等待用户自行修复
